# Анализ рисков и проблем: Переход на Event-Driven архитектуру

## 1. Текущая архитектура (AS-IS)

### Взаимодействия сервисов

| Отправитель | Получатель | Протокол | Периодичность | Данные |
|---|---|---|---|---|
| core-app | ins-product-aggregator | REST (sync) | Каждые 15 мин | Продукты и тарифы |
| ins-comp-settlement | ins-product-aggregator | REST (sync) | 1 раз в сутки (ночью) | Продукты и тарифы |
| ins-comp-settlement | core-app | REST (sync) | 1 раз в сутки | Оформленные за день страховки |
| ins-product-aggregator | 5 страховых компаний | REST/SOAP/GraphQL | На каждый запрос | Тарифы и продукты |

### Проблемы

1. **Синхронная цепочка вызовов с нарастающей задержкой**
   - core-app → ins-product-aggregator → 5 (скоро 10) страховых компаний
   - Время ответа = `max(timeout_insurance_1, ..., timeout_insurance_N)` + сетевые задержки
   - При добавлении 5 новых страховых компаний latency увеличится кратно
   - Один медленный или недоступный партнёр блокирует весь запрос

2. **Polling с фиксированным интервалом**
   - core-app обновляет данные раз в 15 минут независимо от того, менялись ли они
   - ins-comp-settlement — раз в сутки, получая потенциально устаревшие данные
   - Нет механизма уведомлений об изменениях

3. **Нет изоляции отказов внешних систем**
   - Ошибка API одной страховой компании приводит к ошибке для core-app
   - Отсутствует graceful degradation

4. **Избыточная нагрузка на ins-product-aggregator**
   - Каждый запрос core-app и ins-comp-settlement вызывает N параллельных запросов к страховым компаниям
   - При пиковых нагрузках (50 RPS поиск) создаётся до 250 RPS (5×50) исходящих вызовов от aggregator
   - При 10 страховых компаниях: до 500 RPS исходящих вызовов

5. **ins-comp-settlement зависит от core-app для получения данных**
   - Раз в сутки выполняет REST-запрос к core-app
   - Если core-app недоступен или перегружен — settlement не может выполнить свою функцию
   - Жёсткая синхронная связка между сервисами

6. **Отсутствие гарантий доставки при временных отказах**
   - При сетевых проблемах или перезапуске сервисов данные теряются
   - Нет retry-механизма с гарантированной доставкой

7. **ins-comp-settlement не имеет собственной копии данных о страховках**
   - Данные запрашиваются синхронно из core-app
   - При формировании отчёта за месяц необходимо много данных — единовременная нагрузка на core-app

## 2. Планируемый рост нагрузки

| Параметр | Текущее значение | После роста (с 10 СК) |
|---|---|---|
| Количество страховых компаний | 5 | 10 |
| Пользовательская нагрузка (поиск) | 50 RPS | Ожидается рост после рекламной кампании |
| Исходящие вызовы aggregator (пик) | 250 RPS | 500+ RPS |
| Задержка aggregator (при 5 СК) | ~2-5 сек | ~4-10 сек (при 10 СК без изменений) |

## 3. Рекомендуемые изменения

### 3.1 Внедрение Event-Streaming (Kafka)

**Какие взаимодействия перевести на Event-Streaming:**

| Текущее взаимодействие | Новый паттерн | Обоснование |
|---|---|---|
| core-app → ins-product-aggregator (REST pull, 15 мин) | ins-product-aggregator → Kafka → core-app (push) | Устранение polling, немедленное получение обновлений |
| ins-comp-settlement → ins-product-aggregator (REST pull, 24ч) | ins-product-aggregator → Kafka → ins-comp-settlement (push) | Устранение polling, актуальные данные |
| ins-comp-settlement → core-app (REST pull, 24ч) | core-app → Kafka → ins-comp-settlement (push) | Асинхронная доставка, слабая связность |

**События (Event Schema):**

1. `ProductTariffUpdated` — публикуется ins-product-aggregator при изменении продуктов/тарифов в страховых компаниях
   - Содержит полный срез данных или diff изменений
   - Подписчики: core-app, ins-comp-settlement

2. `PolicyIssued` — публикуется core-app при оформлении новой страховки
   - Содержит данные о полисе (ID, тип, сумма, клиент, страховая компания, дата)
   - Подписчики: ins-comp-settlement

3. `PolicyCancelled` — публикуется core-app при отмене страховки
   - Подписчики: ins-comp-settlement

### 3.2 Паттерн Transactional Outbox

**Где применить:**

| Сервис | Операция | Таблица Outbox |
|---|---|---|
| ins-product-aggregator | При получении новых данных от страховых компаний | `outbox_product_events` |
| core-app | При оформлении/отмене страховки | `outbox_policy_events` |

**Как работает:**
1. Бизнес-операция и запись в outbox-таблицу выполняются в одной транзакции БД
2. Отдельный процесс (CDC или фоновый job) читает outbox-таблицу
3. Публикует события в Kafka
4. Удаляет или помечает обработанные записи (после подтверждения Kafka)

**Зачем:**
- Гарантирует атомарность бизнес-операции и публикации события
- Исключает потерю событий при сбоях
- Обеспечивает exactly-once или at-least-once доставку

### 3.3 Сохранение REST для Command-запросов

Следующие взаимодействия остаются синхронными (REST), так как требуют немедленного ответа:

- **Фронтенд → core-app**: оформление страховки, отображение данных пользователя
- **core-app → client-info**: получение/сохранение данных клиента в процессе оформления
- **core-app → Платёжный сервис**: проведение оплаты

## 4. Преимущества предложенного решения

1. **Устранение каскадных задержек** — ins-product-aggregator публикует события асинхронно, core-app не ждёт ответа всех страховых компаний
2. **Слабая связанность сервисов** — ins-comp-settlement перестаёт зависеть от core-app
3. **Изоляция отказов** — проблемы с API одной или нескольких страховых компаний не влияют на core-app
4. **Актуальность данных** — подписчики получают изменения немедленно, без polling
5. **Масштабирование** — Kafka обеспечивает буферизацию при пиковых нагрузках
6. **Гарантированная доставка** — Transactional Outbox + Kafka гарантируют, что события не будут потеряны
7. **Снижение нагрузки на aggregator** — данные публикуются один раз, а не запрашиваются многократно

## 5. Риски перехода

| Риск | Вероятность | Влияние | Митигация |
|---|---|---|---|
| Необходимость управления Kafka (операционная сложность) | Средняя | Среднее | Использовать Managed Kafka (Yandex Managed Kafka) |
| Временная несогласованность данных (eventual consistency) | Высокая | Низкая | core-app и так работал с локальным кешем; консистентность в рамках 1-5 секунд |
| Дублирующаяся обработка событий | Средняя | Низкая | Идемпотентность обработчиков |
| Миграция данных при внедрении outbox | Средняя | Средняя | Поэтапный rollout: сначала publish events, затем подписка |
